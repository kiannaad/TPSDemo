# 角色控制器相关物理学习笔记

## 1. 结论
这轮物理学习不应该以“学完物理”作为目标，而应该以“为自研角色物理控制器建立最小可用的物理认知和代码索引”为目标。

当前最值得掌握的主线不是流体、布料或软体本身，而是以下几类能力：
- 运动学基础：位置、速度、加速度、步长
- 固定步长模拟与子步进
- 碰撞检测与碰撞响应的区别
- 约束思维：先预测，再把状态拉回合法区域
- 角色控制器常见交互：地面、墙、坡道、弹床、击退、移动平台
- 碰撞检测分层：宽相位与窄相位

一句话概括：角色控制器更接近“受约束的运动规则系统”，而不是“完全交给刚体自然演化的真实物理”。

## 2. 先建立的最小物理认知

### 2.1 步长是什么
步长不是“走多远”，而是“这次模拟向前推进了多少时间”。

典型代码：

```csharp
pos += vel * dt;
```

这里：
- `vel` 决定单位时间内移动多快
- `dt` 决定这次模拟覆盖多少时间
- `vel * dt` 才是本次真正移动的距离

### 2.2 为什么要做子步进
子步进不是插值，而是把一个大时间步拆成多个小时间步重新计算。

本质是：
- 一个大步：只更新一次状态，误差更粗
- 多个小步：中途状态会被反复更新，结果更稳定

它的作用主要是：
- 降低积分误差
- 降低碰撞漏检概率
- 提高约束稳定性

但它不是万能的。高速运动下，只减小步长仍然可能穿透，需要配合连续碰撞检测或 sweep/cast。

### 2.3 积分误差是什么
积分误差是：程序用离散小步去逼近连续运动时产生的误差。

最典型来源：
- 步长太大
- 中途状态变化快
- 发生碰撞或约束修正

所以角色控制器里经常会出现：
- 接地抖动
- 下坡不稳
- 高速穿透
- 墙边卡顿

### 2.4 为什么游戏里常用逐步模拟，而不是只用重力公式
重力公式适合“整段时间内条件不变”的问题。

游戏里真实情况通常是：
- 中途撞墙
- 中途接地
- 中途跳跃
- 中途切状态
- 中途受到击退

一旦中途规则变化，就不能靠一个公式从头算到尾，只能按小步推进，并在过程中不断检查和修正。

## 3. 碰撞的基本认识

### 3.1 编程里的碰撞是什么
编程里的碰撞，本质上是：
- 判断两个对象是否已经重叠
- 判断一个对象移动过程中会不会打到另一个对象
- 判断碰到后该怎么修正位置、速度或触发事件

所以它可以拆成两层：
- 碰撞检测
- 碰撞响应

### 3.2 碰撞检测与碰撞响应的区别
碰撞检测解决：
- 有没有碰到
- 可能什么时候碰到
- 接触法线是什么

碰撞响应解决：
- 位置怎么修正
- 速度怎么改
- 是否触发特殊状态或事件

### 3.3 为什么很多碰撞只改法线方向速度
碰撞本质上只约束“不能继续朝接触面里面挤压”。

所以常见处理是：
- 法线方向速度被修正
- 切线方向速度保留

角色撞墙后沿墙滑动，本质上就是这个思路。

## 4. 角色控制器里“物理”的真实角色
角色控制器不是纯物理模拟，更接近：

`运动学驱动 + 碰撞约束 + 状态机 + 手感参数`

它的目标通常不是“真实”，而是：
- 稳定
- 可控
- 可调
- 手感一致

所以角色控制器要处理的不是单一的重力问题，而是：
- 地面怎么站稳
- 墙怎么挡住你但允许滑动
- 坡道怎么投影移动方向
- 台阶怎么迈上去
- 弹床怎么给特殊速度
- 敌人重击怎么击退角色

## 5. 角色控制器常见碰撞交互清单

### 5.1 基础约束类
用于维持角色合法运动状态：
- 地面：接地、落地、允许跳跃
- 墙：阻止法线方向继续前进，保留切线方向滑动
- 天花板：阻止向上穿透，清理向上速度
- 坡道：判断可行走坡度、做移动投影
- 台阶：在一定高度内做迈步修正

### 5.2 速度改写类
用于改变角色当前运动：
- 弹床：给角色向上的速度
- 敌人重击：给角色击退速度或冲量
- 移动平台：把平台运动叠加给角色
- 传送带/风场：给持续速度偏移

### 5.3 事件触发类
用于触发玩法逻辑，不一定阻挡运动：
- 陷阱
- 存档点
- 门
- 传送区

## 6. 代码阅读中的关键示例与价值判断

### 6.1 最值得直接学习的模块

#### 05 Simulate Constraints
作用：演示“先积分，再把状态拉回约束上，再回填速度”。

价值：高。

原因：
- 非常接近角色控制器里的“贴地、不能穿墙、沿表面移动”思维
- 能建立位置修正式约束的直觉

#### 11 Find Overlaps / SpatialHashing
作用：建立邻域查询和空间划分思维。

价值：中高。

原因：
- 对大量对象的附近查询很有帮助
- 对角色本体不是核心，但对周围单位、攻击命中、局部交互有帮助

#### 23 Sweep And Prune Collision Detection
作用：演示宽相位碰撞检测，减少没必要的对象对检查。

价值：中高。

原因：
- 对“碰撞检测分层”非常有帮助
- 对角色控制器的直接帮助，不如地面/墙/坡面处理那样直接
- 更像碰撞系统工程素养课

结论：值得认真看，但不要把它误判成角色控制器主线核心。

#### _XPBD
作用：提供小型刚体 + 约束 + 关节框架。

价值：高。

原因：
- `Integrate -> Solve -> FixVelocities` 这套时序非常值得学
- 让你理解“预测 -> 修正 -> 回填”的组织方式
- 比单章示例更接近可复用内核

### 6.2 了解思路即可的模块

#### 06 Triple Pendulum
核心：节点链 + 距离约束 + 子步进 + 双精度。

价值：中。

原因：
- 能帮助理解“约束系统的误差与精度问题”
- 多摆题材本身不是角色控制器主线

#### 10 Soft Body Simulation
核心：四面体体积网格 + 边长约束 + 体积约束。

价值：中。

原因：
- 能帮助理解复杂约束系统怎么扩展到 3D 体积物体
- 对角色控制器直接落地帮助有限

#### 14 Cloth Simulation
核心：布料顶点 + 拉伸约束 + 弯曲约束。

价值：中偏低。

原因：
- 仍然是在练约束系统思维
- 对角色控制器直接帮助不大

### 6.3 当前阶段可跳过的模块

#### 17 Eulerian Fluid Simulator
核心：网格速度场、压力投影、对流。

价值：低。

原因：
- 对角色控制器主线几乎没有直接帮助

#### 18 FLIP Water Simulator
核心：粒子 + 网格混合液体求解。

价值：低。

原因：
- 很高级，但和当前目标距离太远

## 7. 水体模拟和 Shader 水体的区别

### 7.1 真水体模拟
真水体模拟是在算“水真的怎么动”。
常见状态包括：
- 速度场
- 压力场
- 粒子位置
- 水面高度
- 密度

特点：
- 能和物体做真实或近真实交互
- 成本高
- 适合物理研究或特定玩法

### 7.2 Shader 水体
Shader 水体大多数时候是在画“水看起来怎么动”。
常见手法包括：
- 顶点波动
- 法线贴图
- 噪声
- Fresnel
- 反射/折射
- 泡沫边缘

特点：
- 主要服务视觉表现
- 通常没有完整流体物理状态
- 便宜，适合大量场景水

### 7.3 对角色控制器的意义
对角色控制器来说，水通常更适合做成：
- 特殊介质区域
- 视觉效果 + 状态切换 + 浮力/阻力规则

通常不需要完整流体模拟。
