# C++ 构建与链接入门笔记

## 1. 这部分是做什么的

这份文档整理了本次讨论中与 C++ 入门最相关的一条主线：源码如何从 `.h` / `.cpp` 经过预处理、编译、汇编、链接，最终变成可运行程序；以及为什么会有声明、定义、头文件拆分、`undefined reference`、重复定义、PowerShell 运行方式这些看起来零散但本质上相互关联的概念。

如果从 Unity + C# 转到 C++，这份文档的核心价值不是让你一次性记住所有术语，而是先建立一个稳定的骨架：

- `#include` 是预处理时的文本展开
- 编译器只看当前翻译单元
- 链接器负责在整个程序范围内匹配符号
- 头文件和源文件拆分，是为了控制编译规模、隔离接口与实现、支持模块化和二进制复用

## 2. 读懂它要先抓哪几个点

先抓住下面 6 个点，后面的概念都会顺起来：

1. `#include` 发生在预处理阶段，本质是把别的文件内容贴进当前源码。
2. 一个 `.cpp` 在 `#include` 全部展开后形成一个翻译单元，编译器只对这一份内容做语义检查。
3. 声明的作用是告诉当前翻译单元“这个名字存在以及该怎么用”，定义的作用是“真正提供实现或实体”。
4. 每个 `.cpp` 通常会编译成一个 `.obj`，里面除了机器码，还记录了定义了哪些符号、依赖了哪些符号。
5. 链接器的工作是把所有 `.obj` 和库中的符号做匹配。
6. `undefined reference` 是“有引用没定义”，重复定义是“同一个外部符号被定义了多次”。

## 3. 核心对象和职责

### 3.1 `#include`

- 作用：把头文件内容插入当前源码。
- 阶段：预处理。
- 不负责：自动把别的 `.cpp` 实现一并编译进来。

最关键理解：

```cpp
#include "http_server.h"
```

不是“运行时导入模块”，而是“编译前先把 `http_server.h` 内容贴进来”。

### 3.2 头文件 `.h`

- 主要放声明、接口、类型定义、模板、短小的 `inline` 函数。
- 目的是让其他翻译单元知道“你提供什么能力”。

典型函数声明：

```cpp
int print(int a, int b);
```

它告诉编译器：

- `print` 存在
- 参数是两个 `int`
- 返回值是 `int`

但它不提供函数体。

### 3.3 源文件 `.cpp`

- 主要放普通函数定义和具体实现。
- 负责产出目标文件 `.obj`。

典型定义：

```cpp
int print(int a, int b) {
    return a + b;
}
```

### 3.4 翻译单元

一个 `.cpp` 加上它 `#include` 展开的所有内容，就是一个翻译单元。编译器只在这个边界内理解语法、检查类型、生成目标文件。

这件事决定了两个重要事实：

- 没看到声明，会在编译期报错。
- 看到了声明但没把定义链接进来，会在链接期报错。

### 3.5 目标文件 `.obj`

`.obj` 是单个翻译单元编译后的中间产物，里面包含：

- 当前翻译单元生成的机器码
- 这个翻译单元定义了哪些符号
- 这个翻译单元引用了哪些外部符号

### 3.6 链接器和符号

链接器眼里的“同一个东西”本质上是符号。

- 定义符号：某个 `.obj` 真正提供了这个函数/变量。
- 引用符号：某个 `.obj` 需要这个函数/变量，但自己不提供。

链接器负责在全程序范围内做两件事：

- 给每个引用找到唯一合法的定义
- 检查有没有缺定义或重复定义

## 4. 主链路

从源码到程序的最小主链路可以记成：

```text
.h / .cpp
-> 预处理
-> 编译
-> 汇编
-> .obj
-> 链接
-> .exe
```

对应理解如下：

- 预处理：展开 `#include`、宏等。
- 编译：对当前翻译单元做语义分析和代码生成。
- 汇编：转成更底层的机器相关表示。
- `.obj`：单个翻译单元的编译结果。
- 链接：把所有 `.obj` 和库拼起来。
- `.exe`：最终可执行文件。

业务上通常把 `main` 看成入口，但真实运行前还会有操作系统加载和运行时初始化。

## 5. 常见问题是怎么产生的

### 5.1 为什么会出现 `undefined reference`

典型情况：

- 头文件里有声明
- 当前 `.cpp` 编译时能看到声明
- 但真正实现所在的 `.cpp` 没有一起参与链接

例如项目中：

- [`server/src/main.cpp`](E:/UnityProgram/FPSResearch/server/src/main.cpp) 调用了 `print(1, 2)`
- [`server/src/http_server.h`](E:/UnityProgram/FPSResearch/server/src/http_server.h) 声明了 `print`
- [`server/src/http_server.cpp`](E:/UnityProgram/FPSResearch/server/src/http_server.cpp) 定义了 `print`

如果只执行：

```powershell
g++ main.cpp -o main
```

那么编译能过，但链接器找不到 `print(int, int)` 的定义，于是报：

```text
undefined reference to `print(int, int)'
```

### 5.2 为什么会出现重复定义

关键点是：编译单元彼此隔离，只对编译阶段成立；链接阶段所有 `.obj` 会汇总成同一个程序。

如果把普通函数定义直接写进头文件，并被多个 `.cpp` `#include`：

- `a.cpp` 的翻译单元得到一份定义
- `b.cpp` 的翻译单元也得到一份定义

各自编译都可能没问题，但链接时会发现同一个外部符号被定义了多次，于是报重复定义。

### 5.3 为什么 `inline`、模板、类内函数经常能放头文件

它们属于特殊情况：

- `inline`：语言允许相同定义在多个翻译单元中重复出现。
- 类内定义的成员函数：通常隐式 `inline`。
- 模板：编译器需要在使用点看到完整定义，才能实例化。

所以它们可以放头文件，但普通非 `inline` 函数定义通常不建议这么做。

### 5.4 `#pragma once` 解决什么问题

它主要防的是：

- 同一个头文件在同一个翻译单元里被重复包含

它不能解决：

- 一个带普通定义的头文件被多个不同 `.cpp` 各自包含后产生的程序级重复定义

## 6. 报错分层的快速判断

以后看到一条 C/C++ 报错，可以先按下面的规则判断层次：

- 找不到头文件：预处理/包含阶段问题
- 名字没声明、类型不匹配、语法错误：编译阶段问题
- `undefined reference` / `unresolved external`：链接阶段问题
- 程序成功启动后崩溃或异常：运行时问题

可以直接记成一句话：

```text
找不到文件 -> 预处理
看不懂代码 -> 编译
找不到实现 -> 链接
跑起来才炸 -> 运行时
```

## 7. 为什么 C++ 要这么拆文件

把头文件和源文件拆开，不是为了形式复杂，而是为了工程可扩展：

- 控制编译规模，支持增量编译
- 分离接口和实现
- 支持多人协作和模块边界
- 支持 `.obj`、静态库、动态库、可执行文件这类二进制组织方式

这也是为什么 C++ 比 Unity + C# 更显式地暴露“构建”这件事。C# 和 Unity 的很多复杂度被运行时、程序集系统和引擎工具链屏蔽了，而 C++ 把这些边界更多留给开发者自己管理。

## 8. 初始化方式入门

### 8.1 为什么 `WSADATA wsaData = {};` 这样写

在 WinSock2 示例里，常见写法是：

```cpp
WSADATA wsaData = {};
```

这表示：

- 定义一个 `WSADATA` 局部变量
- 用值初始化的方式把它置为干净的默认状态

对这类 C 风格结构体来说，实际效果通常可以理解成“先整体清零”，然后再把它交给系统 API 填充。

这样写的主要价值是：

- 避免局部变量保留未初始化的脏值
- 让输出结构体在 API 只部分写入时仍然保持可预期状态
- 让代码意图更明确，一眼就知道这个对象是准备给 API 当输出参数用的

### 8.2 常见初始化写法怎么理解

下面这些写法是 C++ 里最容易混淆的一组：

```cpp
int a;
int a = 0;
int a{};
int a = {};

MyStruct s;
MyStruct s{};
MyStruct s = {};
```

可以按下面的实战规则理解：

- `int a;`
  只定义，不初始化。函数内的局部基础类型通常是未初始化状态，不安全。

- `int a = 0;`
  明确初始化为 `0`。

- `int a{};`
  值初始化。对基础类型来说通常也是初始化为 `0`。

- `int a = {};`
  对基础类型来说也通常初始化为 `0`。

- `MyStruct s;`
  如果是局部变量，成员通常不会自动变成 0，可能是未初始化状态。

- `MyStruct s{};`
  对整个对象做值初始化。对于普通结构体，可以近似理解成把成员初始化到干净的默认状态。

- `MyStruct s = {};`
  对普通结构体通常和 `s{}` 的效果接近。

### 8.3 现阶段可以采用的简单规则

如果你是从 C# 过来，当前阶段最重要的不是背术语，而是建立下面这个习惯：

- 局部变量不要默认认为会自动初始化
- 想要“明确值”，就直接赋值
- 想要“干净默认状态”，优先用 `{}` 或 `= {}`

例如：

```cpp
int count{};
bool ok{};
WSADATA wsaData{};
sockaddr_in addr{};
```

这类写法在系统 API、网络编程和结构体初始化里都很常见，也比较稳妥。

## 9. `a.cpp` 为什么还要 include `a.h`

如果某个 `.cpp` 有对应的头文件，那么在 `.cpp` 里 include 这个 `.h` 通常是有明确意义的，不是重复劳动。

核心作用是：

- 强制实现和声明保持一致
- 让头文件的依赖问题尽早暴露
- 保证当前实现文件使用的是同一份接口定义

例如头文件里声明：

```cpp
bool RunHttpHelloServer(unsigned short port);
```

而 `.cpp` 里真正实现这个函数。如果 `.cpp` 不 include 自己的头文件，就可能出现：

- 头文件改了，但实现忘了同步
- 参数类型、返回值、命名空间不一致
- 头文件自身缺依赖，但一直没暴露

因此很多团队会要求：

- `xxx.cpp` 的第一行先 include 自己的 `xxx.h`

这样如果头文件不自洽，或者实现和声明不一致，会在最早的阶段暴露出来。

## 10. 为什么函数不先声明就会报错

### 10.1 规则本身是什么

在 C++ 里，一个函数、类型或变量如果要在某个位置被使用，编译器在读到这一行之前，必须已经知道它的声明。

这意味着：

- 不一定非要把函数声明写到 `.h`
- 但一定要保证“使用点之前，这个名字已经可见”

例如：

```cpp
void Foo()
{
    Bar();
}

void Bar()
{
}
```

上面这段对 C++ 来说通常会报错，因为编译器读到 `Foo()` 里的 `Bar()` 时，还没见过 `Bar` 的声明。

可以用两种方式解决：

```cpp
void Bar();

void Foo()
{
    Bar();
}

void Bar()
{
}
```

或者直接把 `Bar` 的定义写在前面。

### 10.2 为什么不放到 `.h` 也可能没问题

如果某个函数只在当前 `.cpp` 文件内部使用，那么它完全可以只在当前 `.cpp` 顶部做前置声明，而不放到头文件。

头文件更适合放：

- 需要被别的 `.cpp` 共享的声明
- 某个模块真正对外暴露的接口

而像 `SendAll`、`LogLastError` 这种只在实现文件内部辅助使用的函数，更适合：

- 只写在 `.cpp`
- 通过前置声明或定义顺序解决可见性问题

### 10.3 `SendAll` / `LogLastError` 报错说明了什么

如果在同一个 `.cpp` 里出现：

```cpp
void HandleClient()
{
    SendAll(...);
}

bool SendAll(...)
{
    LogLastError(...);
}

void LogLastError(...)
{
}
```

那么会出现：

- `HandleClient` 调用 `SendAll` 时，`SendAll` 还不可见
- `SendAll` 调用 `LogLastError` 时，`LogLastError` 也还不可见

这类报错本质上不是“链接问题”，而是编译器按顺序读取源码时，名字还没建立可见性。

## 11. 为什么 C# 不太需要关心这个，而 C++ 要关心

### 11.1 C# 的体验为什么不同

在 C# 里，你更常见的是按类组织代码，编译器在分析一个类时，通常会先收集这个类的成员信息，所以类内方法之间的位置往往没那么敏感。

例如：

```csharp
class A
{
    void Foo()
    {
        Bar();
    }

    void Bar()
    {
    }
}
```

这通常没有问题。

### 11.2 C++ 为什么更强调声明顺序

C++ 的默认编译模型更接近：

- 头文件先展开
- 编译器按当前顺序一边读一边判断
- 读到某个名字时，如果还没声明，就不能假设“后面也许会有”

所以 C++ 比 C# 更依赖：

- 显式声明
- 文件顺序
- 头文件可见性
- 前置声明

这和 C++ 整体更底层、更显式的构建模型是一致的。

### 11.3 C++ 类成员是不是也要关心位置

要分情况。

对于普通自由函数，位置和声明顺序更敏感。

对于同一个类里的成员函数，通常没那么敏感。例如：

```cpp
class A
{
public:
    void Foo()
    {
        Bar();
    }

    void Bar()
    {
    }
};
```

这通常是可以的，因为编译器在类定义阶段会先知道这个类有哪些成员。

但 C++ 类也不是完全“不关心顺序”，因为它仍然要求：

- 用到的类型要可见
- 类外定义成员函数时，类本身要先声明
- 基类、成员类型、模板参数都要满足可见性要求

### 11.4 一个够用的对照结论

- C#：更像先建立完整类信息，再分析成员调用关系
- C++ 自由函数：更像读到哪里就要求声明先到哪里
- C++ 类成员：比自由函数宽松，但仍要满足类型和声明可见性

如果从 C# 转过来，最应该建立的新习惯是：

- 不默认编译器会“替你看到后面的东西”
- 先思考当前这一行依赖的名字是否已经可见

## 12. Windows PowerShell 里的运行方式

### 8.1 为什么 `main.exe` 直接敲会报错

在 PowerShell 中，当前目录默认不在命令搜索路径里。所以当前目录中存在 `main.exe` 时，直接输入：

```powershell
main.exe
```

仍然可能提示“无法识别”。

正确方式是显式写：

```powershell
.\main.exe
```

这里的 `.\` 表示“当前目录下的”。

### 8.2 `build.ps1` 如何运行

在项目根目录可以执行：

```powershell
.\server\bin\build.ps1
```

如果脚本执行策略拦住，也可以用：

```powershell
powershell -ExecutionPolicy Bypass -File .\server\bin\build.ps1
```

### 8.3 本项目当前的构建脚本做了什么

[`server/bin/build.ps1`](E:/UnityProgram/FPSResearch/server/bin/build.ps1) 目前会：

- 自动收集 [`server/src`](E:/UnityProgram/FPSResearch/server/src) 下所有 `.cpp`
- 一次性调用 `g++` 编译和链接
- 生成 [`server/bin/main.exe`](E:/UnityProgram/FPSResearch/server/bin/main.exe)
- 链接 `-lws2_32`，为 WinSock 相关代码准备系统库

## 13. 常见误区

- 误区：`#include` 会自动把别的 `.cpp` 的实现带进来。
  实际：它只展开头文件文本，不会替你链接实现文件。

- 误区：既然每个翻译单元隔离，就不该有重复定义。
  实际：编译阶段隔离，链接阶段会汇总到整个程序。

- 误区：声明没什么用，不生成代码就等于没参与构建。
  实际：声明会参与编译器的语义分析、类型检查和外部符号引用记录。

- 误区：`inline` 的重点只是性能优化。
  实际：更关键的语言层作用是允许相同定义在多个翻译单元中合法出现。

- 误区：局部变量会像 C# 那样自动初始化。
  实际：C++ 里很多局部变量如果不主动初始化，可能就是未定义的脏值。

- 误区：`.cpp` include 自己的 `.h` 没意义。
  实际：这样做是在用编译器校验“实现是否真的符合头文件里承诺的接口”。

- 误区：函数不放在 `.h` 里就一定不能用。
  实际：关键不是“放不放 `.h`”，而是“使用之前有没有声明”。

- 误区：C++ 里所有函数都像 C# 类成员那样，不关心书写位置。
  实际：普通自由函数更依赖声明顺序；类成员通常更宽松，但依然受类型可见性约束。
