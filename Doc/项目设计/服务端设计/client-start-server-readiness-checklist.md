# 客户端正式开工前的服务端验收标准

## 1. 结论

正式开始写 Unity 客户端前，服务端不需要已经做成完整联机系统，但必须先达到一条清晰的最低门槛：

- 会话可信
- 房间可信
- 重连可信
- 接口稳定
- 主链路可独立验证

换句话说，客户端开工的前提不是“服务端功能很多”，而是“服务端已经能稳定充当权威真源”。

当前阶段真正需要做稳的是最小在线闭环：

`login -> create_room -> join_room -> heartbeat -> reconnect`

只要这条链稳定，客户端就可以正式开始做登录、房间、联调和断线重连主链路。

## 2. 要解决的问题

在正式开发客户端前，最容易出现两个误区：

- 误区 1：觉得必须先把服务端做得非常完整，客户端才值得开始
- 误区 2：服务端只做几个假接口回包，就急着往 Unity 里接

这两个方向都不对。

前者会让项目长期停在“准备阶段”。
后者会让客户端开发建立在不稳定地基上，后面不断返工。

正确做法是先把服务端做到“足以托住客户端开发”的程度，再进入客户端联调和功能开发。

## 3. 服务端开工门槛

下面这 10 条就是正式开始写 Unity 客户端前，服务端应该达到的最低验收标准。

### 3.1 服务端能稳定启动，且启动失败可定位

通过标准：

- 有固定启动方式
- 启动后监听端口明确
- 启动失败时有清晰错误信息
- 不存在“有时能起、有时不能起，但原因不明”的情况

### 3.2 `login` 接口已经跑通，且字段语义稳定

通过标准：

- 能收到登录请求
- 能返回成功响应
- 返回关键字段至少包括 `player_id`、`session_id`
- 失败时能返回明确原因

### 3.3 `Session` 生命周期真实存在

通过标准：

- 登录后服务端会真正创建 session
- 后续请求能根据 `session_id` 识别玩家身份
- 非法或失效 `session_id` 会被拒绝
- 过期逻辑清晰

### 3.4 `create_room` 会创建真实房间

通过标准：

- 调用后服务端内部确实创建 room
- 返回 `room_id`
- 房主身份明确
- 房间成员初始状态正确

### 3.5 `join_room` 能真实改变成员关系

通过标准：

- 第二个玩家能加入已有房间
- 服务端成员列表真实更新
- 重复加入、非法房间、无效 session 会正确拒绝
- 返回结果能体现成员变化

### 3.6 `heartbeat` 真正维护在线状态

通过标准：

- 心跳请求能刷新 `last_heartbeat`
- 超时 session 会进入离线处理
- 离线判定规则明确
- 心跳成功和失败都能从日志确认

### 3.7 `reconnect` 能恢复原有上下文

通过标准：

- 断线后可以重新认证
- 能恢复到原玩家上下文
- 能恢复当前房间或明确告知恢复失败
- 恢复失败路径有清晰返回

### 3.8 服务端已经形成最小权威模型

通过标准：

- 至少有 `Session`
- 至少有 `Room`
- 至少有 `Player` 或等价的玩家状态对象
- 至少有 `match_state` 或等价比赛状态

### 3.9 主链路可以脱离 Unity 独立验证

通过标准：

下面这条链可以不用 Unity，直接通过 Postman 或 `curl.exe` 验证：

1. `login`
2. `create_room`
3. 第二个玩家 `login`
4. `join_room`
5. `heartbeat`
6. `reconnect`

每一步都能确认：

- 请求发出去了
- 服务端收到了
- 服务端内部状态变了
- 返回字段符合预期

### 3.10 主链路日志足够支撑联调

通过标准：

- 能看到请求进入日志
- 能看到 session 创建或校验日志
- 能看到 room 创建日志
- 能看到 player 加房日志
- 能看到 heartbeat 更新日志
- 能看到 reconnect 成功或失败日志

## 4. 通过这 10 条后，可以开始做什么

只要服务端通过上面的 10 条，客户端就可以正式开始开发这些模块：

- 登录界面和登录流程
- Unity 侧 HTTP 请求封装
- Session 保存和续期
- 房间界面
- 房间成员展示
- 断线重连流程
- 进入战斗前的联机主链路

这时继续只埋头补服务端，收益已经开始下降，应该进入“服务端最小闭环 + 客户端同步开发”的节奏。

## 5. 当前阶段不要求先做完的内容

下面这些内容很重要，但不是当前卡住客户端正式开工的前置条件：

- WebSocket 实时同步
- `InputCommand / PlayerState / WorldSnapshot`
- Hitscan 判定
- Damage / Death 广播
- protobuf
- Steam 正式鉴权
- 数据库持久化
- 复杂反作弊
- 分布式拆服
- 线上运维体系

这些属于下一阶段的实时同步和工程化问题，不属于现在的最小开工门槛。

## 6. 推荐推进顺序

建议按照下面的顺序推进，而不是先把服务端无限做深：

1. 先把 `Session + Room + Heartbeat + Reconnect` 做稳
2. 立即开始 Unity 客户端的登录、房间、重连主链路
3. 等这条主链联调稳定后，再进入 FPS 实时同步闭环

## 7. 一句话判断标准

当服务端满足下面这句话时，就可以开始正式动客户端：

**不开 Unity，也能稳定跑通联机主链路；开了 Unity，客户端只是在消费一个已经成立的服务端事实。**
