# 主链路分析与核心概念拆解笔记

## 1. 这部分是做什么的

这份文档的目标不是解释某个具体函数怎么写，而是回答两个更基础的问题：

1. 学服务端和客户端通信时，为什么一定要建立“主链路”。
2. 遇到 `Session`、`Room`、`Heartbeat` 这类概念时，应该怎么从“现实名词误解”走到“工程概念理解”。

结论先说：

- 主链路分析的抽象层次，通常高于单纯的数据流分析。
- 数据流在回答“数据怎么走”，主链路在回答“一个用户目标如何被系统完成”。
- 抽象概念不是空词，它应该被拆成：系统角色、解决的问题、最小状态、行为、参与时机。

## 2. 读懂主链路要先抓哪几个点

如果脑子里只有模块名、函数名、数据结构，很容易“局部都看懂一点，但系统为什么这样跑还是不清楚”。

主链路分析的作用，就是把下面这些东西串起来：

- 用户目标
- 业务步骤
- 模块职责
- 数据变化
- 代码落点

可以把它理解成一个比时序、控制流、数据流更高一层的分析框架。

建议把分析层次分成这 5 层：

1. 用户目标
   例如：登录、建房、加房、开始游戏、重连。
2. 业务步骤
   例如：提交身份、校验身份、创建会话、返回结果。
3. 职责分配
   例如：客户端负责收集输入和显示结果，服务端负责会话和房间状态。
4. 数据与状态
   例如：生成 `session_id`、写入房间成员列表。
5. 代码落点
   例如：哪个文件、哪个函数、哪个 map。

## 3. 主链路分析法

推荐按下面顺序分析，而不是一上来扎进控制流：

### 3.1 从用户目标开始

先问：用户到底想完成什么？

例如：

- 点击登录
- 创建房间
- 进入房间
- 开始游戏

主链路必须从真实用户动作开始，而不是从某个底层函数开始。

### 3.2 拆最少业务步骤

以“登录”举例：

1. 客户端收集身份信息
2. 客户端发起登录请求
3. 服务端校验请求
4. 服务端创建会话
5. 服务端返回登录结果
6. 客户端进入已登录状态

### 3.3 给每一步分配职责

例如：

- 客户端负责：输入、组包、发请求、展示结果
- 网络层负责：传输请求和响应
- Session 模块负责：创建和维护会话
- Room 模块不参与登录

### 3.4 再分析数据变化

例如登录时：

- 输入：`steam_id`、`display_name`
- 服务端生成：`player_id`、`session_id`
- 服务端新增：一条 session
- 客户端保存：登录态

### 3.5 最后才看代码实现

这时再回到：

- 哪个文件
- 哪个函数
- 调用了谁
- 改了哪个字段

这样代码不再是一堆孤立细节，而是在“填写链路表格”。

## 4. 核心概念和职责

下面用统一模板来拆 `Session`、`Room`、`Heartbeat`。

模板是：

- 它是什么系统角色
- 它解决什么问题
- 它最小需要什么状态
- 它有哪些行为
- 它在主链路里的参与时机

### 4.1 Session

#### 它是什么系统角色

`Session` 是“一个已登录连接在服务端上的会话状态”。

它不是“用户本人”，也不是“网络连接对象本身”，而是服务端用来标识当前登录上下文的一份状态。

#### 它解决什么问题

它解决的是：

- 这个请求是谁发的
- 这个客户端当前是否已登录
- 这个连接当前处于什么状态
- 后续请求该归属于哪个玩家

#### 它最小需要什么状态

典型最小状态通常包括：

- `player_id`
- `session_id`
- `steam_id`
- `display_name`
- `current_room_id`
- `last_heartbeat`

#### 它有哪些行为

- 创建会话
- 查找会话
- 刷新会话存活时间
- 销毁会话
- 重连恢复会话

#### 它在主链路里的参与时机

它通常在“登录成功后”产生，并在后续几乎所有联网行为里参与：

- 建房
- 加房
- 心跳
- 重连

### 4.2 Room

#### 它是什么系统角色

`Room` 是“组织一组玩家关系和状态边界的业务对象”。

它不是现实中的房间，而是系统里用来描述“哪些玩家当前属于同一组”的边界。

#### 它解决什么问题

它解决的是：

- 谁和谁在同一局/同一组
- 谁是房主
- 当前房间里有多少玩家
- 后续开始游戏、同步状态时，作用域在哪里

#### 它最小需要什么状态

最小状态通常可以是：

- `room_id`
- `owner_player_id`
- `members`

如果以后扩展，还可能加：

- `state`
- `max_player_count`
- `ready_state`

#### 它有哪些行为

- 创建
- 加入
- 离开
- 解散
- 房主转移
- 开始游戏

#### 它在主链路里的参与时机

它通常出现在“登录成功之后”的多人组织流程里：

- 创建房间
- 加入房间
- 离开房间

### 4.3 Heartbeat

#### 它是什么系统角色

`Heartbeat` 是“维持在线状态和连接有效性的机制”。

它不是一个独立业务对象，更像一个贯穿会话管理的保活机制。

#### 它解决什么问题

它解决的是：

- 服务端怎么知道客户端还活着
- 多久没消息可以认为掉线
- 会话什么时候该过期
- 房间里离线成员何时该清理

#### 它最小需要什么状态

它通常至少依赖：

- `session_id`
- `last_heartbeat`
- 超时阈值

#### 它有哪些行为

- 客户端定时发送心跳
- 服务端刷新最后活跃时间
- 服务端扫描超时会话
- 超时后清理 session 和房间关联

#### 它在主链路里的参与时机

它通常在“登录成功后”开始参与，贯穿整个在线生命周期：

- 登录后保活
- 房间内保活
- 掉线检测
- 重连前的状态判断

## 5. 常见误区

### 5.1 只从现实名词去理解工程概念

例如把“房间”只理解成现实里的房间，会导致对代码里的状态结构和职责感到困惑。

更好的方法是：

先问它解决什么系统问题，再看它保存什么字段。

### 5.2 一上来只看控制流

如果只盯着函数跳转，很容易知道“代码跑过哪些文件”，但仍然不知道：

- 为什么要有这个模块
- 这一步在整条业务链里起什么作用

### 5.3 把抽象概念等同于数据结构

例如认为房间“就是一个 list”，会丢失它作为业务边界的意义。

数据结构只是概念的实现，不是概念本身。

## 6. 建议继续追的文件或问题

如果后续继续结合项目或 demo 学习，建议优先拿下面这些链路继续练：

1. 登录主链路
   `客户端点击登录 -> 服务端创建 session -> 客户端保存登录态`
2. 建房主链路
   `客户端 create_room -> 服务端创建 room -> 返回成员列表`
3. 加房主链路
   `客户端 join_room -> 服务端更新 members -> 客户端刷新房间`
4. 心跳主链路
   `客户端 heartbeat -> 服务端刷新 last_heartbeat -> 超时清理`

以后每分析一个新概念，都可以复用下面这个模板：

```text
【概念名】
它是什么系统角色？

【它解决的问题】
为什么系统需要它？

【它最小状态】
它至少要保存什么数据？

【它的行为】
它有哪些动作？

【它在主链路里的位置】
它在什么时机会参与？
```
