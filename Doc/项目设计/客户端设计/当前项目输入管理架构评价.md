## 当前项目输入架构概览

当前项目已经不是直接在角色控制器里读取键盘、鼠标，而是搭了一层输入管理架构：

```text
Unity New Input System
-> 自动生成的 PlayerInput
-> InputContainer
-> InputHandle
-> InputManager
-> PlayerInputState / VehicleInputState
-> 后续角色控制器或载具控制器消费
```

核心对象职责如下：

```text
PlayerInput.cs
    自动生成类，包装 .inputactions 资源。

InputManager
    创建 PlayerInput，创建不同输入类型的 Handle，负责默认启用 Player Map，支持 Player / Vehicle 切换。

InputHandle
    对外暴露统一操作：Enable、Disable、EnableAction、DisableAction、GetState、AddCallbacks。

InputContainer
    绑定具体 ActionMap，提供整组启停、单个 Action 启停、每帧 Tick。

PlayerInputContainer / VehicleInputContainer
    从 New Input System 的 Action 中读取值，并生成项目自己的输入状态结构。

PlayerInputState / VehicleInputState
    输入快照。它们把设备输入转换成业务语义输入。
```

这个结构的方向是对的：项目代码没有让角色直接依赖 `Keyboard.current.wKey` 这类设备细节，而是把输入先转成了 `MoveInput`、`LookInput`、`JumpPressed`、`SprintHeld`、`AimHeld` 等业务语义。

## 和 UE 输入架构的对应关系

UE 的输入管理可以粗略理解为：

```text
设备输入
-> Enhanced Input
-> InputAction
-> InputMappingContext
-> PlayerController / InputComponent
-> Pawn / Character
```

你的 Unity 项目当前可以对应成：

```text
设备输入
-> Unity New Input System
-> InputAction
-> InputActionMap
-> InputManager / InputHandle
-> 未来的 Character / Vehicle 控制器
```

两者最像的地方是：

```text
UE InputAction          ~= Unity InputAction
UE InputMappingContext  ~= Unity InputActionMap
UE PlayerController     ~= 当前项目还未完整成型的玩家控制权层
UE Pawn / Character     ~= 未来消费 PlayerInputState 的角色实体
```

UE 官方文档里强调：`PlayerController` 表示玩家意志，复杂场景下更适合由 `PlayerController` 处理输入，再向 `Pawn` 发出命令；`Pawn` 可以被替换，`PlayerController` 则更持久。这个思想对当前项目很有参考价值。

当前项目的 `InputManager` 已经有一点 `PlayerController` 的味道：它独立于具体角色，统一持有输入源，并生成输入状态。但它现在更像“输入系统管理器”，还不是完整的“玩家控制权管理者”。

## 做得好的地方

第一，已经有输入快照层。

`PlayerInputContainer.Tick()` 每帧生成 `PlayerInputState`：

```text
MoveInput
LookInput
FirePressed
FireHeld
JumpPressed
SprintHeld
AimHeld
```

这非常适合 locomotion。移动和视角是连续值，跳跃和开火按下是瞬时事件，冲刺和瞄准是持续状态。这个分类比“所有输入都用回调”更适合做流畅运动控制。

第二，已经用 ActionMap 表达控制上下文。

项目里有：

```text
Player
Vehicle
UI
```

并且 `InputManager.SwitchMap(from, to)` 已经体现了“切换控制方案”的想法。这个和 UE Enhanced Input 里通过 Mapping Context 管理步行、载具、UI 等上下文的思路一致。

第三，提供了 Action 粒度禁用接口。

`InputHandle.EnableAction<TAction>()` 和 `DisableAction<TAction>()` 可以用于禁用某个具体动作，例如禁止跳跃、禁止开火。这个方向比在业务代码里到处写键位判断更干净。

第四，角色控制器没有直接依赖 New Input System。

虽然目前还没看到角色消费 `PlayerInputState`，但从结构上看，角色后续可以只依赖项目自己的输入状态，而不必知道 `InputAction`、`ReadValue`、`WasPressedThisFrame` 等 API。这有利于后续接 AI、网络同步、回放和测试。

## 主要问题

### 1. 输入管理器可能还没有真正进入运行时主循环

`InputManager` 继承 `IManager`，并依赖 `GameManager.Update()` 调用每个 manager 的 `Update()`。

但当前 `GameManager` 不是 `MonoBehaviour`，它的 `Update()` 是普通实例方法；`GameLauncher.Update()` 里调用 `_gameManager.Update()` 的代码也是注释状态。这样一来，`InputManager.Update()` 可能不会被实际驱动，`InputContainer.Tick()` 也就不会每帧刷新输入状态。

这是当前架构里最需要优先确认的问题。输入架构设计得再好，如果 Tick 没有稳定跑起来，`PlayerInputState` 就不会成为可靠输入源。

### 2. UI 类型已经出现在枚举里，但没有对应 Handle

`InputType` 里有：

```text
Player
Vehicle
UI
```

生成类 `PlayerInput.cs` 里也有 `UIActions`，但 `InputManager.Init()` 只创建了：

```text
Player
Vehicle
```

这意味着调用 `GetHandle(InputType.UI)` 或 `SwitchMap(..., InputType.UI)` 时会出问题。当前架构已经预留了 UI，但实现还没补齐。

### 3. SwitchMap 只支持一关一开，缺少上下文栈或优先级

UE Enhanced Input 的 Mapping Context 可以动态添加、移除并设置优先级。这样 UI、角色、载具、瞄准、建造模式等上下文可以组合。

当前项目的 `SwitchMap(from, to)` 是简单互斥模型：

```text
Disable(from)
Enable(to)
```

这适合第一版的 Player / Vehicle 切换，但不适合更复杂的情况，例如：

```text
角色移动仍然允许，但打开径向菜单时屏蔽开火。
UI 打开时保留鼠标 Look 或只屏蔽 Move。
瞄准状态叠加一套灵敏度和输入限制。
```

第一版可以保留互斥切换，但需要意识到：它还不是 UE Mapping Context 那种可叠加、有优先级的上下文系统。

### 4. 还缺少 PlayerController / Possess 这层概念

UE 里 `PlayerController` 持有玩家意志，`Possess` 决定当前控制哪个 `Pawn`。这能解决“输入源”和“被控制对象”分离的问题。

当前项目只有：

```text
InputManager -> 输入状态
```

但还没有明确的：

```text
玩家控制权对象 -> 当前受控角色 / 当前受控载具
```

所以现在输入状态生成以后，后续由谁消费、如何切换消费对象、角色死亡或上车后如何转移控制权，还没有稳定落点。

### 5. 禁用动作的层级还需要分清

当前支持 `DisableAction`，这是输入系统层面的禁用。但不是所有限制都应该通过禁用 Action 实现。

例如：

```text
打开菜单 -> 禁用 Player Map
上车 -> Player Map 切到 Vehicle Map
眩晕 -> 可以在业务层屏蔽 Move / Fire
体力不足 -> 业务层不允许 Sprint
武器冷却 -> 战斗层不响应 FirePressed
```

如果所有规则都挤到 `DisableAction`，输入层会逐渐知道太多角色状态。更合理的方式是分层：

```text
控制上下文不可用 -> 禁用 ActionMap
单个输入语义不可用 -> 可禁用 Action
角色状态不允许执行 -> 业务层过滤输入语义
```

## 对流畅 locomotion 的评价

当前 `PlayerInputState` 对 locomotion 是友好的。

移动和视角使用 `Vector2` 连续值：

```text
MoveInput
LookInput
```

这适合每帧输入快照，也适合后续做：

- 加速度。
- 减速度。
- 空中控制。
- 地面转向。
- 相机相对移动。
- 动画 BlendTree 参数。

跳跃使用 `JumpPressed`，冲刺使用 `SprintHeld`，瞄准使用 `AimHeld`，这种瞬时和持续的区分也合理。

建议后续保持这条数据流：

```text
InputManager.Tick()
-> PlayerInputState
-> PlayerController / 当前控制权层
-> CharacterController / Locomotion
-> 物理层
-> 动画层
```

不要让 locomotion 直接读 `InputAction`，也不要让输入层直接修改角色速度。

## 建议的演进方向

### 第一阶段：确认输入 Tick 主链路

先确保 `InputManager.Update()` 每帧稳定执行。否则输入状态不会刷新。

这一点优先级最高，因为它决定当前架构是否真的能工作。

### 第二阶段：补齐 UI Handle 或暂时移除 UI 枚举

如果当前阶段不做 UI 输入，可以暂时避免暴露 `InputType.UI`。如果要保留，就补齐 `UIInputContainer` 和对应 `UIInputState`。

架构上不建议让枚举出现一个无法实际使用的类型。

### 第三阶段：增加玩家控制权层

建议引入类似 UE `PlayerController` 的项目层对象，例如：

```text
PlayerControlController
```

它不负责读取设备输入，而是负责：

- 从 `InputManager` 获取当前输入状态。
- 保存当前受控对象。
- 把输入状态转发给角色、载具或其他可控制对象。
- 处理 Possess / UnPossess 风格的控制权切换。

这样 `InputManager` 继续保持“输入源管理”，而控制权对象负责“玩家意志发给谁”。

### 第四阶段：建立输入过滤层

在输入状态和 locomotion 之间增加一层规则过滤：

```text
Raw PlayerInputState
-> InputFilter / CharacterStateFilter
-> Effective PlayerInputState
-> Locomotion
```

这样可以根据角色状态屏蔽某些语义，例如体力不足不能冲刺、硬直不能移动、换弹中不能开火。

这层不应该关心具体键位，只处理业务语义。

### 第五阶段：再考虑上下文栈或优先级

第一版 `SwitchMap(from, to)` 可以继续用。等出现复杂 UI、瞄准模式、建造模式、菜单叠加时，再考虑把互斥切换升级为：

```text
PushContext
PopContext
SetContextPriority
```

这会更接近 UE Enhanced Input 的 Mapping Context 优先级模型。

## 总体评价

当前输入管理架构方向是正确的，而且已经抓住了几个关键点：

- 用 New Input System 处理设备差异。
- 用 ActionMap 表达控制上下文。
- 用输入快照服务 locomotion。
- 用 Handle 封装启停和读取。
- 用枚举避免大部分魔法字符串。

它现在的问题不是方向错，而是还处在“输入模块独立成型，但还没接入完整玩家控制链路”的阶段。

如果用 UE 架构来评价：当前项目已经有了类似 `InputAction / MappingContext` 的部分，也有一点 `PlayerController` 的输入管理影子；但还缺真正的 `PlayerController -> Pawn` 控制权关系，以及类似 Enhanced Input Mapping Context 优先级的上下文组合能力。

短期最应该做的是：确认 Tick 主链路、补齐或收敛 UI、引入控制权层、让角色控制器只消费输入快照。这样这套输入架构就能很好地支撑后续流畅 locomotion、载具切换、UI 控制和多角色控制。
