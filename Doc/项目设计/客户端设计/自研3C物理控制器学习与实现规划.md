# 自研 3C 物理控制器学习与实现规划

文档类型：项目设计
关联模块：客户端 3C 角色控制器
决策状态：规划中

## 1. 这份文档是解决什么问题的

这份文档的目标，不是教会完整物理引擎，而是帮助把“第三人称角色运动控制器”拆成一个可学习、可实现、可验证的模块。

当前的核心问题不是 Unity 组件不会用，而是：

- `CharacterController` 太黑盒
- `Rigidbody` 太偏通用刚体模拟
- 想要的是一套自己能解释、能调参、能逐步扩展的角色运动模型

因此本轮学习对象应明确为：

**第三人称 3C 中的自研角色物理控制器。**

## 2. 一句话学习目标

不是为了掌握所有物理知识，而是为了自己做出一个 UE 风格的 character motor 骨架，支持：

- 自定义移动速度
- 自定义加速度和减速度
- 自定义摩擦和刹车
- 胶囊体地面适配
- 坡面行走
- 下坡贴地
- 地面态 / 空中态切换

## 3. 为什么不要把目标定成“学完物理”

角色控制器并不是完整刚体模拟问题，而是一个“受约束的运动求解问题”。

刚体系统更关注：

- 力
- 质量
- 惯性
- 自然演化

角色控制器更关注：

- 输入能否稳定转成移动
- 碰到地形后行为是否可预测
- 参数是否可设计
- 手感是否稳定

因此这轮学习的重点不应该是全面学习物理，而应该是：

- 角色控制器需要哪些最小物理与几何知识
- 这些知识如何落到一个可用的 motor 上

### 4.1 主看内容

- 水平运动与垂直运动分离

- 地面吸附 ground snap
- 地面态 / 空中态切换

### 4.2 暂时不看内容

- 完整刚体动力学

如果现在把这些一起看，会导致学习范围失控。

## 5. 你必须先建立的核心认知

### 5.1 角色控制器的本质

角色控制器的核心不是“让物理引擎自己算”，而是：

1. 根据输入得到期望移动意图
2. 根据当前状态和环境得到可执行速度
3. 根据碰撞结果修正实际位移
4. 更新接地状态、坡度状态、空中状态

所以它本质上是：

**探测 -> 求解 -> 移动 -> 修正 -> 更新状态**

### 5.2 角色控制器和刚体的区别

刚体关心的是“物理上自然不自然”，角色控制器关心的是“玩家感觉是否稳定、可控、可解释”。

因此你自研控制器时，不应该默认自己是在复刻真实世界，而是在实现一套游戏规则。

## 6. 必学的 8 个核心知识点

### 6.1 速度与位移更新

需要真正吃透：

- `velocity += acceleration * dt`
- `position += velocity * dt`

这是所有运动求解的基础。

### 6.2 水平速度和垂直速度分离

角色控制里不能把所有速度一股脑混在一起。

至少应分清：

- 平面速度
- 重力速度

这样后面处理上坡、下坡、跳跃、贴地时才不会混乱。

### 6.3 地面法线

“碰到地面”不够，必须知道：

- 地面朝哪边倾斜
- 当前坡是否可行走
- 当前法线是否应被视为墙面

### 6.4 向量投影

你必须熟练使用：

- 把输入方向投影到地面切平面
- 把速度沿碰撞法线方向剔除
- 把速度转成沿墙滑动

### 6.5 胶囊体探测

角色有体积，因此不能只靠 raycast。

必须理解：

- sphere cast
- capsule cast
- overlap
- penetration

### 6.6 Ground Snap

下坡问题的关键通常不是重力本身，而是“刚离地就被判成 airborne”。

因此需要做：

- 当角色离地面很近时，允许继续吸附到地面
- 让角色在可行走坡面上保持稳定贴地

### 6.7 坡度分类

地形接触至少要分：

- Walkable Ground
- Steep Slope
- Wall
- Ceiling

通常依据是法线与世界上方向夹角。

### 6.8 碰撞滑动

角色撞墙或撞到不可行走坡面时，不应该直接卡死，而应该：

- 去掉法线方向速度分量
- 保留切线方向滑动能力

## 7. 推荐的模块拆分

建议你一开始就按模块来写，而不是先写一个超级大脚本。

### 7.1 `CharacterMotor`

职责：

- 对外提供每帧 `Tick`
- 串起 probe、solver、collision、state update

### 7.2 `MovementState`

职责：

- 保存当前速度
- 保存 grounded / airborne 状态
- 保存地面法线、坡度、当前移动模式

### 7.3 `GroundProbe`

职责：

- 地面采样
- 坡度检测
- 近地面吸附判定
- 基础墙面/天花板探测

### 7.4 `MovementSolver`

职责：

- 地面加速度
- 空中加速度
- 摩擦
- 刹车
- 重力
- 坡面方向投影

### 7.5 `CollisionSolver`

职责：

- sweep
- slide
- depenetration
- 多次迭代修正位移

### 7.6 `CharacterConfig`

职责：

- 统一管理参数
- 支持手感调参
- 让设计参数和代码逻辑分离

## 8. 推荐的关键参数

至少先有下面这些参数：

- `MaxWalkSpeed`
- `GroundAcceleration`
- `GroundFriction`
- `BrakingDeceleration`
- `AirAcceleration`
- `Gravity`
- `MaxFallSpeed`
- `JumpSpeed`
- `MaxWalkableSlopeAngle`
- `GroundSnapDistance`
- `StepHeight`
- `CapsuleRadius`
- `CapsuleHeight`

这套参数就是后面“自己设计移动手感”的基础。

## 9. 推荐的分阶段实现路线

### 阶段 1：平地运动闭环

只做：

- 平地移动
- 启动 / 加速 / 减速
- 朝向旋转
- 不接动画
- 不处理复杂地形

这一阶段的目标是先彻底掌握速度控制和手感调参。

通过标准：

- 能清楚解释每个参数对启动和刹车的影响
- 能稳定控制平地移动手感

### 阶段 2：Grounded 判定与地面探测

只做：

- Ground Probe
- 地面法线
- 坡度角输出
- grounded / not grounded 判定

通过标准：

- 能稳定输出当前地面法线和坡度角
- 能解释为什么某个表面被判定为地面或墙面

### 阶段 3：坡面行走与贴地

只做：

- 输入方向投影到坡面
- 下坡 snap to ground
- 陡坡识别
- walkable slope / steep slope 区分

这是最关键的阶段。

通过标准：

- 上坡不跳
- 下坡不漂
- 坡面切换时 grounded 稳定

### 阶段 4：空中态与跳跃

只做：

- Falling 模式
- Jump 起跳
- 空中加速度
- 落地恢复

通过标准：

- 地面态和空中态逻辑清楚分离
- 落地切换稳定，没有明显抖动

### 阶段 5：滑墙、台阶和修正

只做：

- 撞墙滑动
- 小台阶处理
- depenetration
- 多次 sweep 修正

通过标准：

- 撞墙不抽搐
- 小障碍不过度卡住
- 胶囊体不会轻易嵌入地形

## 10. 推荐的调试输出

这个模块必须做可视化调试，否则很难真正学透。

至少要实时显示：

- 当前移动模式
- 当前速度
- 平面速度
- 垂直速度
- Grounded 状态
- Ground Normal
- Slope Angle
- Snap 状态
- 当前探测命中点

同时建议画出：

- 地面法线 debug line
- 胶囊体 probe 范围
- 期望位移方向
- 修正后位移方向

## 12. 参考 UE 时应该学什么

参考 UE，不是照抄所有实现，而是重点学习这些设计思想：

- `MovementMode` 分层
- Ground 和 Air 分开处理
- 主循环是“探测 -> 计算 -> 移动 -> 修正”
- 参数化手感，而不是写死常量

你的第一版最少应该建立：

- `Walking`
- `Falling`

两个模式。

## 13. 这轮最合理的起步任务

不要从复杂地形开始，也不要从完整碰撞系统开始。

最合理的起步任务是：

**第一周先做“平地胶囊体运动 + 加速度 / 摩擦 / 刹车参数系统”。**

这一步完成后，你再进入坡面和贴地，思路会清楚很多。

## 14. 一句话总结

你这轮真正要学透的，不是“Unity 物理组件怎么调”，而是：

**如何把玩家输入、环境探测、运动求解和碰撞修正组织成一个属于你自己的 character motor。**
