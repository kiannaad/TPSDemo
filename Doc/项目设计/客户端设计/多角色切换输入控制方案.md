# 多角色切换输入控制方案

文档类型：项目设计
关联模块：客户端输入管理
决策状态：草案

## 问题背景

在多角色切换控制里，一个常见问题是：如果玩家一直按住 `W` 键，当前角色自然会向前移动；当玩家在按住 `W` 的过程中切换到另一个角色时，新角色是否应该继续向前移动？

答案应该是：应该继续前进。

原因是 `W` 键的按住状态不应该属于某一个角色，而应该属于公共输入模块。切换角色时，变化的不是输入状态本身，而是这份输入状态被谁消费。

## 核心结论

多角色切换的核心不是让每个角色分别监听键盘和鼠标，而是建立一个统一的输入入口，由它持续维护当前输入快照，再由控制权管理者决定这份输入快照发送给哪个角色。

也就是说：

```text
切换角色不是切换输入状态，而是切换输入快照的消费对象。
```

对应的数据流是：

```text
设备输入 -> 输入快照 -> 当前控制目标 -> 具体语义映射 -> 当前角色业务回调
```

## 输入快照

输入快照表示当前这一帧玩家输入的完整状态。它不是某个角色的状态，而是玩家输入源的状态。

例如：

```csharp
public struct PlayerInputSnapshot
{
    public float MoveAxisForward;
    public float MoveAxisRight;
    public bool JumpDown;
    public bool FireHeld;
    public bool FireDown;
}
```

当玩家按住 `W` 时，公共输入模块每帧都能读到：

```text
MoveAxisForward = 1
```

如果这时切换角色，下一帧新的当前角色仍然会收到这份移动输入，所以它会继续向前移动。旧角色停止接收玩家输入，因此旧角色不再继续响应玩家控制。

## 输入分类

输入需要分成两类处理：持续状态输入和瞬时事件输入。

### 持续状态输入

持续状态输入表示“当前是否正在发生”的输入，比如：

- `WASD` 移动。
- 鼠标视角轴。
- 按住开火。
- 按住瞄准。

这类输入在角色切换后可以继续被新角色消费。

例如玩家按住 `W` 切换角色：

```text
按住 W -> 旧角色向前移动 -> 切换角色 -> 新角色下一帧继续向前移动
```

### 瞬时事件输入

瞬时事件输入表示“这一帧刚刚发生”的输入，比如：

- 跳跃按下。
- 换弹按下。
- 交互按下。
- 切换武器按下。

这类输入只应该在真正按下的那一帧触发，不应该因为切换角色而补触发。

例如玩家按住 `Space` 后再切换角色，新角色不应该凭空跳一次。只有玩家重新按下 `Space`，新角色才应该收到新的跳跃事件。

## 设计规则

角色本身不直接监听键盘和鼠标。键鼠监听应该集中在公共输入模块里。

公共输入模块负责从设备输入生成输入快照。它不关心当前是谁在执行动作。

控制权管理者负责保存当前受控角色，并决定输入快照发送给谁。切换角色时，只修改当前控制目标。

当前角色消费玩家输入，执行移动、跳跃、开火、换弹等具体业务回调。

非当前角色不消费玩家输入。后续如果需要让非当前角色行动，可以交给 AI 或命令系统处理，但不要让 AI 或每个角色各自监听玩家键鼠。

## 示例流程

玩家按住 `W` 并切换角色时，流程应该是：

```text
1. 公共输入模块读取设备输入，得到 MoveAxisForward = 1。
2. 控制权管理者把输入快照发送给当前角色 A。
3. 角色 A 向前移动。
4. 玩家按下切换键。
5. 控制权管理者把当前控制目标从角色 A 改为角色 B。
6. 玩家仍然按住 W，公共输入模块下一帧继续得到 MoveAxisForward = 1。
7. 控制权管理者把新的输入快照发送给角色 B。
8. 角色 B 向前移动，角色 A 停止响应玩家输入。
```

这个方案能够保证持续输入在角色切换后自然延续，同时避免跳跃、换弹、交互这类瞬时事件在切换时误触发。

## 和角色控制器架构的关系

这个方案仍然符合当前角色控制器的整体思路：

```text
输入层 -> 物理层 -> 动画层
```

输入层负责把设备信号转成玩家意图，并决定当前由哪个角色消费这份意图。

物理层消费移动、跳跃、蹲伏等输入语义，产出角色的空间变化和运动状态。

动画层再根据物理层产出的数据修正动作表现。

因此，多角色切换不应该破坏输入层、物理层、动画层的分工。它只是在输入层内部增加了一个“当前控制目标”的选择关系。
