# 最小 3C 客户端设计方案

## 1. 结论

客户端当前阶段的目标，不是一次性做完整 3C，而是先做出一个能够稳定支撑后续联机开发的最小闭环：

- 第三人称相机先用 Cinemachine
- 输入层单独封装，不直接耦合运动和动画
- 运动层优先建立独立的 `PlayerMotor`
- 动画层参考 UE 的思路，动画不驱动位移，只消费运动结果
- 射击层先做相机中心射线命中 + 上半身开火表现

第一版要解决的不是“表现多完整”，而是“运动事实、动画表现、射击判定”三者的边界清楚。

## 2. 当前阶段要解决的问题

当前客户端开工最容易出现两个问题：

- 运动、动画、相机、射击全堆在一个控制脚本里，后面难以扩展
- 一开始就把动画或物理组件当成主控制器，导致后面多人同步和手感调参都很痛苦

因此当前设计必须优先满足：

- 控制逻辑可解释
- 参数可调
- 动画和位移解耦
- 后续可以逐步替换 `PlayerMotor` 的底层实现，而不推翻整套客户端结构

## 3. 整体分层

建议客户端 3C 最小骨架拆成 5 层。

### 3.1 输入层 `PlayerInputBridge`

职责：

- 读取输入系统
- 输出稳定的高层输入意图
- 不直接修改位移和动画

建议输出字段：

- `Vector2 MoveInput`
- `Vector2 LookInput`
- `bool SprintHeld`
- `bool AimHeld`
- `bool FirePressed`
- `bool JumpPressed`

### 3.2 运动层 `PlayerMotor`

职责：

- 维护角色物理状态
- 计算实际速度和位移
- 输出给动画层和射击层消费的运动结果

建议输出字段：

- `Vector3 WorldVelocity`
- `Vector3 LocalVelocity`
- `float PlanarSpeed`
- `bool IsGrounded`
- `float YawDelta`
- `float AimPitch`

这一层是客户端 3C 的核心，后续即使替换成自研控制器，也应尽量保持对外接口稳定。

### 3.3 动画层 `PlayerAnimatorDriver`

职责：

- 把 `PlayerMotor` 的结果转成 Animator 参数
- 控制 Locomotion、Aim、Fire 等动画状态
- 保证动画只负责表现，不负责真实位移

建议输入参数：

- `MoveX`
- `MoveY`
- `Speed`
- `Grounded`
- `Aim`
- `Fire`
- `Pitch`

### 3.4 射击层 `PlayerCombatController`

职责：

- 根据输入和相机进行最小射击判定
- 提供开火事件给动画和特效系统
- 后续可逐步扩展成真正武器系统

当前阶段先做：

- 准星
- 屏幕中心射线
- 命中点计算
- 开火反馈

### 3.5 表现层 `PlayerView`

职责：

- 模型、枪口、特效、音效、UI 挂点管理
- 接收运动和战斗事件做纯表现

## 4. 运动层设计原则

当前阶段的关键原则只有一句：

**真实移动由运动层决定，动画层只消费运动结果。**

这意味着：

- 输入不是位移
- 动画也不是位移
- `PlayerMotor` 才是角色空间状态的唯一真源

这样设计的好处：

- 手感调参更清楚
- 后续接服务端权威同步更自然
- 动画资源和运动逻辑不会绑死

## 5. 动画层设计

动画层参考 UE 常见的 locomotion 组织方式，但落地到 Unity Animator。

### 5.1 Base Layer

职责：下半身基础移动。

建议状态：

- `Idle`
- `Locomotion`
- `Airborne`

其中 `Locomotion` 第一版建议直接做 2D Blend Tree，参数为：

- `MoveX`
- `MoveY`

基础采样至少包括：

- Forward
- Backward
- Left
- Right
- Idle

### 5.2 UpperBody Layer

职责：上半身瞄准和开火覆盖。

建议状态：

- `Empty`
- `AimPose`
- `FireOverlay`

配合 Avatar Mask，只覆盖脊柱以上部位。

这样可以实现：

- 下半身继续跑动
- 上半身保持瞄准
- 射击动作只叠加到上半身

### 5.3 Root Motion 策略

当前阶段不建议让 Root Motion 驱动角色真实位移。

建议：

- 移动动画全部使用 in-place
- 胶囊体或后续自研 motor 负责真实位移
- Root Motion 以后只考虑给特殊动作用，例如翻越、处决、冲刺起步

## 6. 射击层最小方案

当前阶段射击层只做最小闭环，不做完整武器系统。

推荐方案：

- 屏幕中心准星
- 以相机中心射线作为命中真源
- 枪口朝向和动画只负责表现
- 命中结果先用 debug hit 或基础受击逻辑验证

这样做的原因：

- 第一版最需要的是“指哪打哪”的基本可用性
- 相机中心射线更利于调试瞄准体验
- 后续再逐步补枪口校正、弹道、后坐力

## 7. 第一阶段实现顺序

建议严格按下面顺序推进。

### 阶段 1：纯运动闭环

只做：

- 第三人称相机
- 胶囊体移动
- 加速度 / 减速度
- 重力
- 朝向旋转

这一阶段不接 Animator。

### 阶段 2：接基础动画

只做：

- Idle
- Locomotion Blend Tree
- Grounded / Airborne 切换

### 阶段 3：接瞄准和开火

只做：

- Aim 状态
- 上半身蒙版
- 开火动画和特效
- 相机中心射线命中

### 阶段 4：调手感

集中调：

- 运动参数
- 旋转速度
- 瞄准时移动衰减
- 动画阻尼
- 相机手感

## 8. 当前阶段明确不做的内容

为了保证范围收敛，下面这些内容当前不纳入第一版客户端 3C：

- 完整武器系统
- 复杂后坐力和程序化摆动
- Root Motion 驱动的完整移动
- 翻越、攀爬、滑铲
- 网络同步
- 复杂状态机和技能系统

## 9. 推荐的第一版验收标准

当下面这些条件都成立时，说明最小 3C 客户端闭环已经成立：

- 胶囊体能在平地稳定移动
- 相机方向和移动方向关系清晰
- 角色在移动和停止时没有明显抖动
- 动画可以根据局部速度平滑切换
- 角色可以边移动边瞄准
- 开火时命中和表现一致

## 10. 这一版设计和后续自研控制器的关系

这份设计并不要求底层一定使用 Unity `CharacterController` 或 `Rigidbody`。

更准确地说，这份文档定义的是客户端 3C 的上层边界：

- 输入如何进入
- 运动层应该输出什么
- 动画层如何消费运动数据
- 射击层如何和相机配合

后续底层从临时 motor 逐步替换成自研控制器时，客户端整体结构不应该被推翻。
